# SimpleRAG-Go 项目规划书

## 1. 项目概述

### 1.1 项目名称

SimpleRAG-Go

### 1.2 项目定位

本项目目标是从零实现一个“简单但完整”的 RAG 系统。系统以 Go 为主语言，面向高并发在线问答场景，强调以下几个原则：

- 架构足够简单，便于个人或小团队快速落地
- 核心链路完整，具备真实生产雏形
- 在线请求路径短，保证吞吐、延迟与稳定性
- 离线索引链路清晰，支持文档持续更新
- 保留后续扩展能力，但不在第一阶段过度设计

### 1.3 项目愿景

实现一个单机即可运行、可观测、可压测、可评测、可持续迭代的 RAG 基线系统，并作为后续多租户、分布式、混合召回增强版本的基础。

## 2. 建设目标

### 2.1 业务目标

- 支持用户上传知识文档并建立检索索引
- 支持用户基于知识库进行问答
- 返回答案时提供可追溯引用片段
- 在常见中小规模知识库下保持稳定响应

### 2.2 技术目标

- Go 实现核心服务，具备较强并发处理能力
- 离线与在线链路解耦
- 支持向量检索、关键词检索与重排
- 提供标准化 REST API
- 提供日志、指标、链路追踪和压测方案

### 2.3 质量目标

- 代码结构清晰，适合长期维护
- 核心接口具备单元测试和集成测试
- 检索与生成效果可量化评估
- 系统具备基本容错、超时、限流与熔断能力

## 3. 项目范围

### 3.1 MVP 范围

- 文档接入：TXT、Markdown、PDF
- 文档清洗
- 文本切片
- Embedding 生成
- 向量存储
- 关键词索引
- 混合检索
- Rerank
- Prompt 构建
- LLM 调用
- 引用片段返回
- REST API
- 管理端基础接口
- 监控日志
- 基础评测与压测

### 3.2 暂不纳入 MVP

- 多租户隔离
- 复杂权限系统
- 在线学习与反馈闭环
- 多模态检索
- 分布式索引服务
- 流式增量重建大规模索引
- Agent 编排

## 4. 总体设计原则

- 先保证正确性，再优化性能
- 先支持单机高并发，再扩展分布式
- 在线路径只做必要操作
- 离线任务异步执行，避免阻塞
- 所有外部调用必须设置超时、重试和熔断策略
- 所有关键数据处理过程需要幂等
- 所有核心指标必须可观测

## 5. 系统架构

### 5.1 架构分层

建议拆分为五层：

1. 接入层
2. 应用层
3. 检索层
4. 数据层
5. 可观测与运维层

### 5.2 核心模块

#### A. Ingest 服务

负责文档上传、解析、清洗、切片、元数据抽取、任务投递。

#### B. Indexer 服务

负责生成 embedding、写入 pgvector、构建全文检索索引、更新文档状态。

#### C. Query 服务

负责查询理解、召回、重排、上下文构建、调用 LLM、返回答案和引用。

#### D. Metadata 服务

负责知识库、文档、切片、任务状态等元数据管理。

#### E. Observability 模块

负责日志、指标、trace、审计、错误告警。

### 5.3 在线/离线链路拆分

离线链路：

- 上传文档
- 解析与清洗
- 切片
- embedding 生成
- 建立向量和关键词索引

在线链路：

- 接收问题
- Query 改写
- 多路召回
- Rerank
- 拼接上下文
- 调用 LLM
- 返回答案与引用

这种拆分能显著降低在线路径复杂度，适合 Go 服务在高并发下做稳定治理。

## 6. 技术选型建议

### 6.1 后端

- Go 1.22+
- HTTP 框架：Gin 或 Echo
- 配置管理：Viper 或纯环境变量
- 数据库访问：pgx
- ORM：不强制，建议少量场景使用 sqlc 或手写 SQL

### 6.2 存储

- PostgreSQL：业务主库
- pgvector：向量存储
- PostgreSQL FTS：关键词检索
- Redis：缓存、热点结果、限流计数、短生命周期任务状态

### 6.3 异步任务

MVP 阶段优先使用库内任务队列：

- PostgreSQL/Redis 驱动的轻量任务机制
- worker pool + 重试 + 死信策略

后续可升级为：

- NATS
- Kafka

### 6.4 模型能力

- Embedding 模型：优先外部 API，后续可替换本地模型
- Rerank 模型：优先外部 API
- LLM：通过统一接口抽象，支持替换不同供应商

### 6.5 可观测

- 日志：结构化 JSON
- 指标：Prometheus
- Trace：OpenTelemetry
- 看板：Grafana

## 7. 核心数据流设计

### 7.1 文档导入流

1. 用户上传文档
2. 系统落库存储文档元数据
3. 解析正文并清洗格式噪音
4. 根据策略切片
5. 生成切片任务
6. 异步生成 embedding
7. 写入向量库和关键词索引
8. 更新文档状态为可检索

### 7.2 问答查询流

1. 接收用户问题
2. 执行基础清洗与可选改写
3. 进行向量召回
4. 进行关键词召回
5. 合并结果并去重
6. 调用 rerank
7. 选出 Top-K 片段
8. 构建 Prompt
9. 调用 LLM
10. 返回答案、引用片段、耗时和召回信息

## 9. 切片策略设计

### 9.1 设计目标

- 保证语义完整度
- 控制上下文长度
- 减少召回碎片化
- 保留来源定位能力

### 9.2 初版策略

- 按段落优先切分
- 目标 chunk 大小：300 到 800 tokens
- overlap：50 到 120 tokens
- 保留标题、页码、段落序号等元信息

### 9.3 后续优化

- 基于标题层级切片
- 针对表格和代码块做专用切片
- 语义边界切片

## 10. 检索策略设计

### 10.1 初版方案

采用混合检索：

- 向量检索负责语义召回
- 关键词检索负责精确术语召回

### 10.2 排序流程

1. 向量 Top-N
2. 关键词 Top-N
3. 合并去重
4. 规则打分
5. Rerank Top-K

### 10.3 关键参数建议

- 向量召回 N：30 到 100
- 关键词召回 N：20 到 50
- 最终重排 K：5 到 12

### 10.4 检索质量风险

- chunk 太小导致语义破碎
- chunk 太大导致噪音增加
- 仅向量召回容易漏掉精确关键词
- 仅关键词召回容易漏掉同义表达

## 11. 生成策略设计

### 11.1 Prompt 原则

- 明确回答边界
- 只允许基于已检索上下文回答
- 无法确认时明确说明不知道
- 保留引用标记

### 11.2 输出格式建议

- answer
- citations
- confidence
- debug_info

### 11.3 幻觉控制

- Prompt 约束来源
- 设置引用片段
- 控制上下文窗口噪音
- 对低相关召回设置拦截阈值

## 12. 高并发设计重点

### 12.1 总体思路

Go 的优势应主要用在请求并发、I/O 复用、任务处理和资源治理上，而不是把复杂模型逻辑堆在应用层。

### 12.2 在线服务治理

- 每个请求使用 `context.Context`
- 数据库连接池限额
- 外部模型调用连接复用
- 热点查询缓存
- 并发召回但总量受控
- 全链路超时传播
- 限流与熔断

### 12.3 异步任务治理

- 固定 worker pool
- 不同任务类型分队列
- 重试退避
- 死信记录
- 任务幂等

### 12.4 关键性能指标

- QPS
- P50 / P95 / P99 延迟
- 检索耗时
- 模型调用耗时
- 索引构建吞吐
- 任务失败率

## 17. 风险与应对

### 17.1 模型成本高

应对：

- 缓存 embedding
- 批量生成 embedding
- 区分在线与离线模型规格

### 17.2 检索效果不稳定

应对：

- 优先打磨切片
- 引入混合检索
- 做小规模标准评测集

### 17.3 高并发下延迟抖动

应对：

- 严格控制连接池
- 做调用超时和隔离
- 减少在线同步步骤

### 17.4 文档质量参差不齐

应对：

- 增强清洗规则
- 增加导入校验
- 标记低质量文档

## 18. 成功标准

满足以下条件可视为 MVP 成功：

- 能稳定导入常见文本类文档
- 能完成端到端 RAG 问答
- 能返回可追溯引用
- 在中等并发场景下保持稳定
- 有基础监控、日志和压测结果
- 代码结构支持后续继续演进

## 19. 里程碑建议

### 里程碑 M2

完成导入、切片、embedding、索引链路。

## 20. 总结

这个项目的关键不是把所有 RAG 技术一次性堆满，而是建立一个结构清楚、性能稳定、可迭代优化的 Go 基线系统。第一阶段最重要的决策是：

- 架构上拆清楚在线与离线
- 检索上采用混合召回
- 工程上优先保障并发治理和可观测
- 产品上坚持“有引用、可回溯、可评测”

如果按本规划推进，最终会得到一个足够简单、但具备真实工程价值的 RAG 起点版本。
